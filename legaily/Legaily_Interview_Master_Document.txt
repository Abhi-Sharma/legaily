Legaily is an AI-powered legal platform that helps lawyers and judges process legal documents faster, search laws intelligently, generate legal drafts, and securely share files. It combines OCR, NLP, RAG, decentralized storage, and role-based dashboards to modernize legal workflows.

- Reduces repetitive legal drafting work from hours to minutes.

SECTION G: BEST CHALLENGES FACED
--------------------------------------------------------------------------------
Challenge 1: Poor Quality Scanned Documents
- Issue: OCR accuracy drops dramatically on blurry, crooked, or poorly lit physical files.
- Solution: Solved using OpenCV pre-processing to apply grayscale, deskewing, and thresholding before text extraction.

Challenge 2: Hallucination in Legal Answers
- Issue: Standard LLMs guess answers, which is dangerous in legal systems.
- Solution: Solved using a strict Retrieval-Augmented Generation (RAG) pipeline grounded specifically on the Indian Penal Code (Using InLegalBERT & FAISS).

Challenge 3: Slow Secure Uploads
- Issue: Decentralized storage (IPFS) is often slower than normal cloud storage.
- Solution: Solved using asynchronous progress indicators and robust UI state management to keep the user informed during cryptographic locking.

SECTION H: METRICS / IMPACT
--------------------------------------------------------------------------------
- Processing time for a 20-page document reduced from ~90 minutes to ~3 minutes.
- Draft generation reduced from ~45 minutes to <5 minutes.
- Legal query response engineered for 99%+ grounded accuracy (no hallucinations).
- Full legal processing workflows compressed from hours to mere minutes.

LAYER 3: INTERVIEW PREPARATION (Q&A)
================================================================================

BASIC QUESTIONS
--------------------------------------------------------------------------------
Q1. Explain Legaily.
A: Legaily is an AI-powered legal platform combining OCR, NLP, RAG, and decentralized storage to help lawyers automate file processing and answer legal queries accurately based on the IPC. (Expand to 1 min pitch if prompted).

Q2. Why did you choose the legal domain?
A: It's an area still heavily dependent on manual paperwork, physical reading, and legacy tools. Applying tech here solves massive real-world bottlenecks.

Q3. Who are your target users?
A: Primary users are Lawyers who need speed and organization. Secondary users are Judges who need a secure way to share confidential case files.

Q4. What main problems does it solve?
A: Manual paperwork extraction, slow legal research, repetitive manual drafting, and insecure, centralized file sharing.

TECHNICAL QUESTIONS
--------------------------------------------------------------------------------
Q5. Why React on the frontend?
A: For a fast, dynamic, SPA (Single Page Application) experience using a Virtual DOM, making module switching seamless without page reloads.

Q6. Why Node.js specifically?
A: Node.js is non-blocking and asynchronous, perfect for handling multiple users uploading heavy documents or querying AI concurrently.

Q7. Why is OCR needed?
A: Most real-world legal documents aren't digital text; they're scanned, physical papers (images/PDFs) that machines can't read directly.

Q8. Why use OpenCV before Tesseract OCR?
A: To clean the noise. Pre-processing images (deskewing, altering contrast) improves OCR accuracy from ~60% to 95%+.

Q9. Why RAG instead of just prompting a plain chatbot?
A: Standard LLMs hallucinate, which is unacceptable in law. RAG anchors the LLM strictly to our IPC database, providing factual, citable answers.

Q10. What is FAISS and why use it?
A: FAISS is a vector database optimized for similarity search. It allows us to search through thousands of embedded IPC statutes in milliseconds based on semantic meaning, not just exact keywords.

Q11. What is InLegalBERT?
A: It's an embedding model trained specifically on Indian legal documents, allowing it to correctly embed legal semantics far better than standard general-purpose models.

Q12. What role does LangChain play?
A: It orchestrates the entire intelligence flow: taking the user prompt, routing it to FAISS, constructing the context-injected prompt, and retrieving formatted output from the LLM.

Q13. Why use IPFS for storage instead of S3/Google Drive?
A: IPFS is decentralized and generates a hash (CID) based on file content. This guarantees the file is tamper-proof—if one word changes, the CID changes entirely.

Q14. What exactly is a CID?
A: Content Identifier. A unique cryptographic string generated by IPFS that points to a specific file on the decentralized network.

Q15. Why use JWT?
A: It's a stateless, highly secure standard for verifying user identity, essential for implementing robust Role-Based Access Control between Lawyers and Judges.

AI & DOMAIN QUESTIONS
--------------------------------------------------------------------------------
Q16. How does the summarization actually work?
A: Text extracted via OCR and translated is passed into the Cohere API, instructed to perform 'abstractive summarization' to extract the key dates, parties, and core legal issues.

Q17. How do you ensure legal answers are accurate?
A: Through a tight RAG implementation. We isolate the context to retrieved IPC sections and use strict prompt engineering: "ONLY use the provided text."

Q18. What if the user asks something not in the IPC?
A: The FAISS search will return low-confidence matches. The prompt instructs the LLM to reply "I cannot find relevant information in the IPC" instead of guessing.

PRODUCT QUESTIONS
--------------------------------------------------------------------------------
Q19. Why is this better than just Googling the law?
A: Google returns SEO blog posts or opinions. Legaily returns the exact section of the IPC and directly contextualizes it to your specific user query.

Q20. Why would a lawyer actually use this?
A: Because it directly buys them time. Converting 90 minutes of reading a Marathi FIR into a 3-minute English summary allows them to take on more cases.

Q21. What is the highest-value feature?
A: Depending on user context, either the RAG-based Legal Query (for rapid, accurate legal research) or DocAI (for massive unstructured document processing).

SCENARIO QUESTIONS
--------------------------------------------------------------------------------
Q22. What if an uploaded document is completely unreadable due to damage?
A: OpenCV and OCR will struggle yielding low text confidence. The UI gracefully handles this by warning the user that the summary may be inaccurate due to poor image quality.

Q23. What if 1,000 lawyers upload files at the exact same time?
A: Express handles requests asynchronously, but for heavyweight tasks (OCR), we would introduce a queue processing system (like BullMQ + Redis) and scale worker nodes horizontally.

Q24. IPFS can sometimes be slow. How did you handle that UX?
A: By keeping the process asynchronous and immediately displaying loading indicators, so the user knows deliberate encryption and decentralized storage are taking place.

OWNERSHIP & EXPERIENCE QUESTIONS
--------------------------------------------------------------------------------
Q25. What was your biggest personal learning in this project?
A: Building RAG pipelines to control LLM hallucinations, applying Web3 concepts directly to real-world security issues, and building robust, asynchronous NodeJS event-driven APIs.

Q26. If you rebuilt Legaily today, what would you do differently?
A: I would shift the heavy OCR/AI processing to a strict microservice architecture communicating via message queues, rather than handling it heavily within the main API instance.

ADVANCED FOLLOW-UP QUESTIONS
--------------------------------------------------------------------------------
Q27. How would you scale this to 1 million users?
A: Implement a CDN for the frontend, move to an event-driven microservices architecture, implement database sharding for case files, rely on managed Vector DBs, and deploy via Kubernetes.

Q28. How would you monitor this in production?
A: Implement robust logging (Winston/Morgan), application performance monitoring (Datadog/NewRelic), and alerting for failed AI or OCR requests to ensure high uptime.

Q29. How would you secure confidential files even further?
A: By encrypting the files (AES-256) locally on the browser BEFORE uploading them to IPFS, ensuring not even the IPFS nodes can see the raw content.

Q30. How would you reduce LLM API costs at scale?
A: I'd aggressively cache common queries using Redis (Semantic Caching), use batch processing, leverage smaller specialized local models where possible, and perform async background jobs.
